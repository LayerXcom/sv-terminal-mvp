SV Terminal MVP System Event Flow

LinearをSVのSSoTにし、Slack相談、n8nイベントキュー、エンジニアPC上のlocal worker、Codex、GitHub PR、本番反映確認をつなぐMVP構成。 SVの操作面は薄く保ち、重い実行はlocal PCのpoll workerに集約する。

Slack

問い合わせ専用channelと経理相談thread。投稿内容はLinear公式syncでissueに蓄積する。

Linear

private team、tenant Project、親issue、phase sub issue、Event Log comment threadを持つ。

👩‍✈️SV Chrome

SV_ACTION marker挿入、proposal承認、周知文承認を補助。SVの操作面は薄く保つ。

n8n

Webhook、routing、audit log、Data Table queue。local workerはここをpollする。

🧑‍💻Engineer PC

localops-linear-bridge。Linear/n8nをpollし、Codex、backtest、PR作成、本番確認を実行。

GitHub

approval_rulesのPR、review、merge。Linear連携でPR状態をissueへ戻す。

エンジニアPCはinbound portを開けず、n8n queueとLinear APIをpollする。

MVP1: Slack問い合わせからルール反映

経理相談をLinearに蓄積し、方針確定後にagentがPR化する。

💬
Slack問い合わせ
SVが専用channelへ投稿。経理相談threadの起点になる。
🔀
n8n bootstrap
Slack eventを受け、Linear作成イベントをqueueする。
◇
Linear issue
tenant Project、親issue、phase sub issue、Event Logを作る。
👩‍✈️
方針確定
SVが相談結果を確認し、SV_ACTION markerでagentへ渡す。
🤖
ルール下書き
local workerがCodexを起動し、proposalをLinearへ投稿。
⑂
PR / Review
approval_rulesを編集し、PR・backtest・reviewを実施。
🚀
本番確認
merge後にrule versionとsmokeを確認し、Linearに記録。
📣
周知 / Close
Slackへ変更宣言。sub issue完了後に親issueをclose。

Linear / n8n / Worker データ設計

イベント本文はLinear/Slackに残し、n8n queueは最小メタデータだけを持つ。

Linear issue

  • private team: SV Ops
  • Project: tenant
  • parent issue: delivery unit
  • sub issues: phases
  • assignee: ball owner
  • labels: trigger / action / resolution / risk

Event Log thread

  • botが親issue作成時に固定commentを作る
  • threadにSV_EVENTを追記
  • proposalやapprovalは別thread
  • 処理結果はSV_ACTION_RESULTで戻す
[SV_EVENT id=evt_001 type=pr_created status=done]

n8n queue

  • event_id
  • linear_issue_id
  • type / status / dedupe_key
  • claimed_by / lease_until
  • attempts / last_error / processed_at
