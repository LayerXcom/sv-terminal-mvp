# インタビュー初回質問

## まず決めたいこと

1. このターミナルが最初に勝つべき瞬間は、どの瞬間か。
   - 月初の大量エスカレを捌く瞬間
   - 経理相談をルール反映まで見失わない瞬間
   - ルール作成 AI / PR / review feedback を人間が監督する瞬間
   - 複数テナントを 1 人の SV が持てると実感する瞬間

2. 「1 件のイシュー」の粒度は何か。
   - 申請 1 件
   - Slack 相談 thread 1 本
   - ルール変更 1 件
   - 顧客/経理との論点 1 件
   - ルール改修から周知までの親 work item 1 件

3. Linear を SSoT にする場合、Linear に持たせたい情報と、外部リンクで十分な情報の境界はどこか。

4. 人間が必ず押したいボタンは何か。
   - 方針確定
   - PR 作成してよい
   - 周知してよい
   - production 反映済みにしてよい
   - 一般ルール化しない
   - 顧客/経理に確認する

5. AI に勝手にやってほしいことと、提案までに留めたいことの境界はどこか。

6. Slack は原本として扱いたいか、ハードコピー thread を作って運用したいか。

7. 最初の対象 workflow は「経理エスカレ -> ルール反映 -> 周知」に絞るべきか、それとも「ルール性能低下の改善」も同時に入れるべきか。

## 回答済み

### Q1. このターミナルが最初に勝つべき瞬間

- 月初の大量エスカレを、SV が破綻せず捌けること。
- 経理相談 -> ルール下書き作成 AI への HITL -> 本番反映までの流れを SV がトラックできること。
- Linear は起票からデリバリーまでのコンテキスト中心として機能させたい。
- 承認ルール更新は、ビジネス要件 -> コード生成 -> PR / CR -> 承認 -> デリバリーという、ソースコードデリバリーと同じ構造で捉える。

## 次に聞くべきこと

Q2. issue / phase / item の粒度をどう切るか。

### Q2 仮回答

親 issue = delivery unit が自然。

delivery unit の起点:

- 経理への相談内容
- オペレーターからのエスカレーション
- Agent からの特定ルールの精度低下アラート

完了条件:

- 1 つないし複数のルールが本番反映される
- 必要に応じて周知される
- AI 精度向上に寄与する形で close できる

phase は issue の中に持つ。issue 数の増加は許容する。
